## 2026-05-26 17:07 KST — 고퀄리티 Color Match 게임 구현 연구

### Title

고퀄리티 Color Match 게임 구현 리서치 및 기술 스택/아키텍처 설계

### Why (목적)

웹 기반 멀티 게임 플랫폼에 입점할 첫 번째 게임으로, 시각적으로 뛰어나고(High-quality) 중독성 있는 Color Match (예: Match-3 또는 SameGame 방식) 게임을 개발하기 위한 구조 및 핵심 로직을 연구한다.

### 1. 게임 메커니즘 및 규칙 (Game Mechanics)

*   **그리드(Grid):** N x M 배열 (예: 8x8) 기반의 보드.
*   **블록/타일:** 5~6가지의 명확히 구분되는 색상(또는 아이콘)으로 구성.
*   **매칭 조건:** 사용자가 인접한 두 타일을 교환(Swap)하여 가로 또는 세로로 3개 이상의 동일한 색상이 연결되면 매치 성공. (Match-3 방식)
*   **연쇄 반응 (Combo/Cascade):** 매치된 블록이 파괴되면, 위의 블록들이 중력에 의해 떨어지고 빈 공간은 새로운 블록으로 채워짐. 이후 다시 매치가 발생하면 콤보(Combo)로 인정되어 추가 점수 부여.
*   **점수 시스템:** 기본 매치 점수 + 콤보 시 배수(Multiplier) 적용.

### 2. 비주얼 및 UX/UI 방향 (Aesthetics)

고퀄리티 느낌을 주기 위해 단순한 색상 블록을 넘어선 시각적 피드백이 필수적이다.
*   **블록 디자인:** 입체감 있는 Glassmorphism 또는 Neon Glow 효과 적용, 모서리 라운딩(`border-radius`).
*   **애니메이션 (핵심):**
    *   **Swap:** 부드러운 위치 교환 (`transition: transform 0.3s ease`).
    *   **Destroy/Match:** 블록 파괴 시 축소 또는 팝(Pop) 효과 (`transform: scale(0)` + 투명도 변화).
    *   **Fall:** 빈 공간으로 떨어지는 바운스 효과.
*   **반응성:** 클릭/터치 시 햅틱/시각적 피드백 (누를 때 약간 축소되는 쫀득한 느낌).

### 3. 기술 스택 및 렌더링 방식 분석

| 렌더링 방식 | 장점 | 단점 | 적합성 평가 |
| :--- | :--- | :--- | :--- |
| **HTML DOM + CSS** | 구현이 빠름, 반응형 웹 적용 쉬움, CSS 트랜지션 활용 용이 | 수천 개의 파티클 구현 시 성능 저하 가능성 | **✅ 채택:** `AGENT.md`의 단순성 원칙에 부합. 순수 CSS 애니메이션만으로도 고퀄리티 구현이 충분히 가능함. |
| **HTML5 Canvas (2D)** | 복잡한 렌더링 및 대량의 객체 처리에 최적화, 성능 우수 | 충돌 처리, 이벤트 리스너(클릭 좌표 계산)를 직접 구현해야 함 | ❌ 초기 구현 복잡도가 불필요하게 높음 |
| **WebGL (PixiJS 등)** | 최고 수준의 그래픽, 하드웨어 가속 완벽 지원 | 외부 라이브러리 의존성 발생, 초기 세팅 및 학습 필요 | ❌ 현 단계에서 오버엔지니어링 우려 |

**결론:** 순수 **Vanilla JS + DOM API + CSS3 (Transitions/Animations)**를 사용하여 게임 상태 관리(Logic)와 화면 렌더링(UI)을 분리한 구조로 빠르게 구현한다.

### 4. 데이터 구조 및 상태 관리 (State Management)

#### 4-1. 핵심 상태 (State)
```javascript
const state = {
  board: [], // 2D 배열 (예: 8x8), 각 셀은 { id, color, row, col } 객체
  score: 0, // 현재 점수
  combo: 0, // 연쇄 매칭 콤보 수
  isAnimating: false, // 진행 중인 애니메이션 유무 (사용자 입력 차단용)
  selectedTile: null // 사용자가 첫 번째로 클릭/선택한 타일
};
```

#### 4-2. 주요 로직 흐름 (Logic Flow)
1.  **Initialize:** 보드(그리드) 생성 및 랜덤 색상 할당. (시작하자마자 매치되는 경우가 없도록 필터링 로직 포함)
2.  **Input (Click/Touch):**
    *   첫 번째 타일 선택 (강조 표시).
    *   두 번째 타일(인접한 타일) 선택.
3.  **Swap:** 두 타일의 데이터 상 위치 교환 및 DOM 업데이트 (애니메이션 대기).
4.  **Check Matches:** 보드 전체를 순회하며 가로/세로 3개 이상 연속된 타일 탐색.
    *   매치가 없으면: 위치 교환을 취소하고 다시 원래 위치로 되돌림 (Swap Back 애니메이션).
    *   매치가 있으면: `isAnimating = true`로 설정하여 추가 입력을 막고 파괴 단계로 진입.
5.  **Destroy & Fall (Cascade):**
    *   매치된 타일 파괴 (점수 및 콤보 증가, UI 업데이트).
    *   파괴된 타일 위의 블록들을 아래로 한 칸씩 이동 (중력 작용).
    *   최상단 빈 공간에 새로운 랜덤 타일 생성 및 낙하 애니메이션.
6.  **Recursive Check:** 블록이 다 떨어진 후 다시 4번 단계(Check Matches)로 재귀하여 연쇄(Combo) 발생 확인. 더 이상 매치가 없으면 `isAnimating = false`, `combo = 0`으로 초기화하여 사용자 입력 대기.

### 5. 기술 스택 분석

| 기술 옵션 | 장점 | 단점 | 판단 |
|-----------|------|------|------|
| **Vanilla HTML/CSS/JS** | 단순, 의존성 없음, 빠른 초기 개발 | 규모 확장 시 관리 어려움 | ✅ 초기 단계에 적합 |

**결론:** 초기에는 **Vanilla HTML + CSS + JavaScript**로 시작하여 빠르게 프로토타입을 구축하는 가장 AGENT.md A-2(단순성 우선) 원칙에 부합한다.

### 6. 디자인 시스템 방향
